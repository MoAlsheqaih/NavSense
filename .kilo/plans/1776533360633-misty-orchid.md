# Integration Plan: MIP Shortest-Path Routing into NavSense Flutter App

## Executive Summary

This plan outlines how to integrate the Python-based MIP routing (Single-Commodity Network Flow) into your existing Flutter application. Currently, your app uses a Dijkstra-based routing algorithm implemented in Dart (`floor_route_datasource.dart`). The Python MIP code provides a different optimization approach that could be integrated via a backend service.

---

## 1. Current Architecture Analysis

### Existing Routing Stack
| Component | Location | Algorithm |
|-----------|----------|-----------|
| Route Service | `lib/services/routing/route_service.dart` | Interface (abstract) |
| Floor Route DS | `lib/data/datasources/floor_route_datasource.dart` | Dijkstra (O(V²)) |
| Mock Route DS | `lib/data/datasources/mock_route_datasource.dart` | Static mock data |
| Route Plan Entity | `lib/domain/entities/route_plan.dart` | `RouteStep`, `RoutePlan` |

### Python MIP Code Features
- **Algorithm**: Single-Commodity Network Flow MIP (PuLP + CBC solver)
- **Graph Building**: Critical nodes + room interior nodes
- **Edge Creation**: K-nearest neighbors with line-of-sight validation
- **Path Finding**: Optimal shortest path with flow conservation constraints

## 2. Integration Options

### Option A: Backend Service (Recommended for MIP)
Deploy the Python MIP solver as a REST API that the Flutter app calls.

**Pros:**
- Leverages CBC solver (not available in Dart)
- Centralized floor plan management
- Easy updates without app release

**Cons:**
- Requires backend infrastructure
- Adds network latency

### Option B: Port to Dart
Rewrite the MIP logic in pure Dart.

**Pros:**
- No network dependency
- Offline capability

**Cons:**
- Dart lacks PuLP/CBC - would need to implement simplex algorithm
- Significant development effort
- May not be truly "MIP" without external solver

### Option C: Pre-compute Routes
Generate all room-to-room routes offline and bundle JSON in app.

**Pros:**
- Fastest runtime (lookup only)
- No solver needed

**Cons:**
- Inflexible to floor plan changes
- Memory for pre-computed paths

---

## 3. Disadvantages of the Current Python MIP Approach

| Disadvantage | Impact |
|--------------|--------|
| **External Solver Required** | CBC solver must be installed on server; adds deployment complexity |
| **Slower than Dijkstra** | MIP overhead vs. O(V²) Dijkstra - the LP relaxation is integral but still needs solver initialization |
| **Less Practical for Mobile** | Not suitable for client-side execution without heavy Flutter/Dart ports |
| **No Real-time Re-routing** | Cannot dynamically adjust to user position during navigation |
| **Hardcoded Floor Plan** | Python code defines rooms/corridors/doors as constants - needs abstraction |

## 4. Implementation Plan: Backend API

### Architecture
```
┌─────────────────┐      HTTP       ┌────────────────────────┐
│   Flutter App    │ ──────────────► │   Python MIP Backend   │
│  (Navigation)   │                │   (FastAPI + PuLP)    │
└─────────────────┘                └────────────────────────┘
        │                                   │
        │ Fallback                          │
        ▼                                  ▼
┌─────────────────┐                ┌────────────────────────┐
```

### Step 3: FastAPI Endpoints

```python
# routes.py
@app.post("/api/route")
async def compute_route(request: RouteRequest):
    result = solver.solve_between_rooms(
        start_room=request.origin,
        end_room=request.destination
    )
    return {
        "path": result['path'],
        "distance": result['distance'],
        "solve_time": result['solve_time']
    }

@app.get("/api/rooms")
async def list_rooms():
    return [{"id": r.id, "name": r.name, ...} for r in rooms]
```

### Step 4: Flutter HTTP Client

1. Add `http` package to `pubspec.yaml`
2. Create `MipRouteDatasource` in `lib/data/datasources/`
3. Update `route_repository_impl.dart` to use MIP endpoint

### Step 5: Offline Fallback

Keep existing `FloorRouteDatasource` as fallback for:
- No network connectivity
- Backend unavailable

## 6. Expected Issues & Mitigations

| Potential Issue | Mitigation |
|-----------------|------------|
| CBC solver installation | Use Docker container with CBC pre-installed |
| Network latency > 3s | Keep Dijkstra as fast fallback |
| Room ID mismatch | Map Flutter room IDs to backend room IDs |
| Visualization not needed | Skip PNG generation, return path only |
